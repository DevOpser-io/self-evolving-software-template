# Mobile release builds (Android + iOS)

The template ships a Capacitor 5 setup: `mobile-chat-app.html` at the repo root is packaged by `./build-mobile.sh` into `frontend/public/static/dist/` and synced into the native projects. This rule covers **signed release** builds — not `npm run dev`. Canonical human docs: [`../../../README.md`](../../../README.md) → **"Mobile Apps (Optional)"**.

## Non-negotiables

- **Never commit `android/keystore.properties`, the keystore file itself, or anything under `ios/App/App/Assets.xcassets/` containing provisioning profiles.** The `.gitignore` already excludes them; do not add exceptions.
- **Never regenerate `android/` or `ios/` in a committed branch.** They're per-developer artifacts generated by `npx cap add <platform>`.
- **Never paste a keystore password into a CI log.** Use the CI provider's secret store; `configure-android-signing.sh` reads from `keystore.properties`, which the CI should generate from secrets at build time and delete after.

## Pre-flight (both platforms)

```bash
node -v      # 20 or 23 LTS
npx cap --version   # Capacitor 5.x
```

Check `MOBILE_ENV` decisions with the user before building:

- `MOBILE_ENV=local` → bakes in `http://localhost:8000`. Useful for on-device testing against the dev laptop.
- `MOBILE_ENV=staging` → requires `STAGING_URL=https://...` set in the shell that runs the build.
- `MOBILE_ENV=production` → requires `PRODUCTION_URL=https://...` set in the shell that runs the build.

`build-mobile.sh` also bakes in the `endpoints` block and app metadata — don't fork the script; edit its substitutions.

## Android release build

### One-time: generate a keystore

```bash
keytool -genkey -v \
  -keystore release.keystore \
  -alias my-key-alias \
  -keyalg RSA -keysize 2048 -validity 10000
```

Store the keystore **outside** the repo (the path in `keystore.properties` below is relative to `android/app/`, so `../../release.keystore` points at the repo root).

### One-time: create `android/keystore.properties`

```
storeFile=../../release.keystore
storePassword=<store-password>
keyAlias=my-key-alias
keyPassword=<key-password>
```

### One-time: patch `android/app/build.gradle`

```bash
./configure-android-signing.sh
```

This reuses the existing script — **do not** write a replacement. It reads from `keystore.properties` and wires the release variant of `android/app/build.gradle` to the keystore.

### Build the release artifact

```bash
PRODUCTION_URL=https://your-domain.com npm run mobile:build:prod
npm run mobile:sync
cd android && ./gradlew assembleRelease     # APK
# or
cd android && ./gradlew bundleRelease       # AAB for Play Store
```

Signed APK lands in `android/app/build/outputs/apk/release/`; signed AAB in `android/app/build/outputs/bundle/release/`.

## iOS release build

iOS signing is **Xcode-only**. Do not try to automate it with CLI scripts unless the user has explicitly asked for fastlane and accepted the complexity.

1. Set the production URL and build the web bundle:

   ```bash
   PRODUCTION_URL=https://your-domain.com npm run mobile:build:prod
   npm run mobile:sync
   ```

2. Open the iOS project:

   ```bash
   npm run mobile:ios
   ```

3. In Xcode:
   - **Signing & Capabilities** → select the Apple Developer team.
   - **Product → Archive** → uploads to Organizer.
   - From Organizer: **Distribute App → App Store Connect** (or Ad Hoc / Enterprise depending on the user's distribution model).

Refer the user to Capacitor's iOS docs for the per-release specifics — don't reinvent the steps.

## Verification gate

Before reporting a release build complete:

- [ ] `./build-mobile.sh` ran clean for the chosen `MOBILE_ENV`, with the matching URL in the environment.
- [ ] `npm run mobile:sync` succeeded.
- [ ] **Android:** `./gradlew assembleRelease` produced an APK whose signing cert matches the keystore alias; `apksigner verify --verbose` shows `Verifies` on the artifact.
- [ ] **iOS:** Xcode Archive succeeded and the Organizer shows the build under the expected bundle identifier.
- [ ] `keystore.properties`, the keystore file, and any provisioning profiles are **not** staged or committed. `git status` is clean of them.

If any is false, do not hand the artifact to the user.
